Unidad 4. Servicio DNS (Domain Name System)

Introducción

El sistema de nombres de dominio DNS (Domain Name System) proporciona un mecanismo eficaz para llevar a cabo la resolución de nombres de dominio a direcciones IP. Como usuarios (humanos) nos es más fácil dirigirnos a un nombre de dominio (de host, de web, de servidor de correo, etc.) utilizando un texto identificativo (por ejemplo, www.gva.es) que a la dirección IP pertinente (por ejemplo, 193.144.127.85). El servicio DNS no sólo permite hacer la resolución de nombres de dominio a direcciones IP, sino también la resolución inversa. Es decir, a partir de una IP averiguar el nombre de dominio.

El servicio DNS proporciona independencia del nombre de dominio respecto a la IP. Así un dominio puede cambiar de IP de forma transparente para los usuarios del dominio. Incluso es usual que un dominio se identifique con más de una IP como medida de redundancia contra la caída del sistema o como balanceo de cargas. Otros servicios proporcionados por el DNS son la identificación de los servidores de correo de un dominio, de cada uno de los hosts que pertenecen a la red, servidores de impresión, etc.

Sistemas de nombres planos y jerárquicos

El problema de la identificación de equipos se produce desde el principio de la existencia de las redes de ordenadores y no es algo específico de TCP/IP. Hacía falta un lenguaje humano para realizar esta identificación.

En los albores de las redes, cuando ARPANET (la red predecesora de Internet), los nombres los equipos se centralizaban en un archivo llamado host.txt (/etc/hosts en Linux), que incluía el nombre del equipo y su IP. Esto es lo que se conoce como un sistema de nombres plano. Puede ser adecuado para redes pequeñas pero no es escalable ni práctico en redes grandes y mucho menos en Internet.

Ejemplo de fichero de nombres plano:


127.0.0.1       localhost
127.0.1.1       hostname
192.168.1.10    servidor
192.168.1.20    pc1

Elementos del sistema de nombres de dominio

El espacio de nombres de dominio está formado por los nombres válidos utilizados para identificar servicios o máquinas en una red. Se puede representar mediante una estructura jerárquica de topología arbórea, es decir, todos los nombres forman un árbol invertido donde cada nodo se separa de los otros nodos por un punto.

Nombres de dominio

Los nombres de dominio pueden estar formados por una o más cadenas de caracteres separadas por puntos y no se distingue entre mayúsculas y minúsculas. Por ejemplo, www.deaw.es. es lo mismo que WWW.deaw.ES..

Los nombres de dominio se expresan como secuencias de etiquetas (labels).

Dominios raíz

En teoría, todos los dominios deben de terminar con un punto (.). Es así porque el árbol de nombres de dominio (espacio de nombres de dominio) empieza con el dominio . que se conoce como dominio raíz (root). En realidad es un elemento nulo de 0 caracteres que se representa con un punto (.).

Un dominio se lee de derecha a izquierda, empezando por el punto ., aunque en la práctica lo hacemos de izquierda a derecha. El punto inicial, generalmente se omite ya que los programas lo añaden por defecto y es meramente formal, pero en ocasiones, será necesario que indiquemos el nombre de dominio completo incluyendo el dominio raíz, es lo que se conoce como nombres de dominio completos (Fully Qualified Domain Names, FQDN).

Dominios y subdominios

Como consecuencia de la organización jerárquica del espacio de nombres de dominios, podemos utilizar los términos dominio y subdominio. Por ejemplo, deaw.es. es un subdominio del dominio es. y www.deaw.es. es un subdominio del dominio deaw.es..

Los dominios o subdominios que cuelgan del dominio raíz . se conocen como dominios de primer nivel o dominios de nivel superior (Top Level Domains, TLD), los que cuelgan de los dominios TLD se denominan dominios de segundo nivel y así sucesivamente.

Zonas

Una zona es una porción del espacio del espacio de nombre de dominio en el DNS cuya responsabilidad administrativa recae sobre un único responsable.

Los servidores que gestionan la zona tienen información completa sobre ella y se dice que son autorizados para esa zona.

Las zonas se almacenan en archivos de texto o en bases de datos, según el tipo de software que se utilice para montar el servidor DNS y de como se configure.

Tomemos como ejemplo el dominio deaw.es. y veamos parte de su archivo de zona:


...
deaw.es.      IN NS    ns1.deaw.es.
ns1.deaw.es.  IN A     192.168.1.20
natos.deaw.es. IN A    192.168.1.21
waor.deaw.es.  IN A    192.168.1.22
www.deaw.es.   IN CNAME natos.deaw.es.
ftp.deaw.es.   IN CNAME waor.deaw.es.
...

A cada una de las líneas del fichero se las conoce como registros de recurso (RR: Resource Records) y definen los tipos de datos en el Domain Name System (DNS). Se utilizan para almacenar datos sobre nombres de dominio y direcciones IP. Una base de datos o fichero de zona está formada por una serie de registros de recursos. Cada registro de recurso da información pertinente sobre un objeto determinado. Por ejemplo, los registros de tipo (A) asocian un nombre de host con una dirección IP, y los registros de puntero de búsqueda inversa (PTR) asocian una dirección IP con un nombre de host y un registro (NS) define un servidor DNS para la zona. El servidor DNS utiliza estos registros de recurso para resolver las consultas de los hosts de su zona.

Cuando un servidor DNS es autorizado para una zona, es el responsable de los nombres de dominio para esa zona. En nuestro ejemplo, ns1.deaw.es es el servidor autorizado para la zona deaw.es. y en él se definen los nombres que cuelgan de deaw.es como por ejemplo, www.deaw.es, ftp.deaw.es, natos.deaw.es, etc.

La organización que administra el servidor DNS y por lo tanto la zona, puede delegar o no alguno de sus subdominios. Supongamos que de deaw.es. cuelgan los subdominios teoria.deaw.es. y practicas.deaw.es. y se decide delegar solo el subdominio practicas.deaw.es.. Esto implica que existirá otro servidor DNS autorizado para el dominio practicas.deaw.es., que almacenará el fichero de zona para dicho dominio.

Una zona no es lo mismo que un dominio. Un dominio es un subarbol del espacio de nombres de dominio y los datos asociados a los nombres de un dominio pueden estar almacenados en una o varias zonas, distribuidas en uno o varios servidores DNS.

Info

Básicamente una zona es una porción de un dominio.

Un servidor DNS puede ser autorizado sobre varias zonas, por ejemplo, el mismo servidor DNS puede ser autorizado para la zona deaw.es. y para la zona seguridadinformatica.es..

Tipos de RR (Resource Record)

En esta subsección vamos a ver cuáles son los registros de recursos o RR más utilizados. Antes debemos aclarar algunos conceptos:

$TTL (Time To Live)

El TTL o tiempo de vida determina, en segundos, durante cuánto tiempo son validos los RR. Pueden indicarse en semanas ($TTL 1W), días ($TTL 7D), horas ($TTL 168H) o minutos (10080M).

En otras palabras, el TTL indica cuánto tiempo tardarán en aplicarse los cambios que le hagamos a un RR desde que los hacemos. En el ejemplo del párrafo anterior, los servidores DNS comprobarán cada semana si se ha producido algún cambio en esos RR. Debe declararse al inicio del archivo de zona.

$ORIGIN

La directiva $ORIGIN define el nombre del dominio que será añadido al final de cualquier nombre que no acabe en punto (nombres relativos o no cualificados) en los RR, para así transformarlos en nombres FQDN (fully qualified domain name). Si un nombre acaba en punto, se considera un nombre FQDN y no se utilizaría $ORIGIN.

Su sintaxis o forma de escribirlo será:

$ORIGIN nombre-dominio

Por ejemplo:


$ORIGIN deaw.es.
; A partir de aquí se añade deaw.es. a todos los nombres relativos

Formato general de los RR

El formato con el que se introducen los RR en los archivos de zona es del siguiente estilo:

Nombre de dominio [TTL] Clase Tipo Tipo-Dato

Así por ejemplo, un RR quedaría tal que así:

profesor.deaw.es 7200 IN A 192.168.10.254

Tipos de registros

Aclarados los puntos anteriores, ahora sí vamos a ver los principales tipos de registros:

  • Registro SOA (Start Of Authority)

    Especifica información autoritaria sobre una zona DNS, incluyendo el servidor de nombre primario, el email del administrador, el número de serial o versión de la zona, y varios temporizadores.

    Ejemplo:

    
deaw.es. IN SOA ns1.deaw.es. super.deaw.es. (
    20190425001 ; serial
    604800      ; refresh (7 días)
    86400       ; retry (1 día)
    2419200     ; expire (28 días)
    604800 )    ; TTL negativo (7 días)
    Explicación del registro SOA
    • deaw.es.
      Indica el nombre del dominio al que pertenece la zona DNS.
    • IN
      Es la clase del registro. IN significa Internet y es la más habitual en DNS.
    • SOA (Start Of Authority)
      Define el registro de autoridad de la zona DNS. Contiene la información principal de configuración y control de la zona.
    • ns1.deaw.es.
      Servidor DNS primario (Master) de la zona. Es el servidor autorizado principal desde el que se propagan los cambios.
    • super.deaw.es.
      Correo electrónico del administrador de la zona. En DNS, el primer punto sustituye a la @, por lo que equivale a: super@deaw.es
    • 20190425001 (Serial)
      Número de versión de la zona. Se incrementa cada vez que se realiza un cambio para que los servidores secundarios sepan que deben actualizarse.
      Formato habitual: YYYYMMDDNN → 2019/04/25, revisión 01
    • 604800 (Refresh)
      Tiempo en segundos que los servidores secundarios esperan antes de comprobar si hay cambios.
      Equivale a 7 días.
    • 86400 (Retry)
      Tiempo que espera un servidor secundario para reintentar la conexión con el primario si falla.
      Equivale a 1 día.
    • 2419200 (Expire)
      Tiempo máximo que un servidor secundario puede seguir usando la zona sin contactar con el primario.
      Pasado este tiempo, la zona se considera no válida.
      Equivale a 28 días.
    • 604800 (TTL negativo)
      Tiempo durante el cual se cachean las respuestas negativas (por ejemplo, cuando un dominio no existe).
      Equivale a 7 días.
      Ejemplo: Si intentas entrar a esta-web-no-existe-12345.com, el servidor DNS responderá con un error llamado NXDOMAIN (Non-Existent Domain).
      • Sin TTL Negativo: Si 1.000 personas escriben mal tu dominio al mismo tiempo, tu servidor DNS recibiría 1.000 consultas inútiles preguntando por algo que no existe.
      • Con TTL Negativo: El servidor DNS le dice a los navegadores: "Este dominio no existe, y no me vuelvas a preguntar por él durante los próximos 7 días, porque la respuesta seguirá siendo la misma".

  • Registro NS (Name Server)

    Cuando se delega la administración de subdominios en otros servidores, este registro indica cuáles son esos servidores autorizados.

    
...
deaw.es. IN NS ns1.deaw.es.  ; Servidor DNS maestro
deaw.es. IN NS ns2.deaw.es.  ; Servidor DNS esclavo
deaw.es. IN NS dns.deaw.net. ; Servidor DNS esclavo

ns1.deaw.es. IN A 192.168.10.20
ns2.deaw.es. IN A 192.168.10.21

; DELEGACIÓN
practicas.deaw.es. IN NS ns1.practicas.deaw.es.
redes.deaw.es.     IN NS dns.deaw.net.
    Explicación de los registros NS y la delegación
    • deaw.es. IN NS ns1.deaw.es.
      Indica que ns1.deaw.es es un servidor DNS autorizado para el dominio deaw.es. En este caso actúa como servidor DNS maestro.
    • deaw.es. IN NS ns2.deaw.es.
      Define a ns2.deaw.es como servidor DNS esclavo del dominio deaw.es, proporcionando redundancia.
    • deaw.es. IN NS dns.deaw.net.
      Establece un servidor DNS externo como servidor autorizado para el dominio deaw.es.
    • ns1.deaw.es. IN A 192.168.10.20
      Registro A que asocia el nombre del servidor DNS ns1.deaw.es con su dirección IP.
    • ns2.deaw.es. IN A 192.168.10.21
      Registro A que asigna una dirección IP al servidor DNS ns2.deaw.es.
    • Delegación de subdominios
      Permite que partes del dominio principal sean gestionadas por otros servidores DNS.
    • practicas.deaw.es. IN NS ns1.practicas.deaw.es.
      Delegación del subdominio practicas.deaw.es a su propio servidor DNS ns1.practicas.deaw.es.
    • redes.deaw.es. IN NS dns.deaw.net.
      Delegación del subdominio redes.deaw.es a un servidor DNS externo.

  • Registro A (Address)

    También conocido como registro de dirección, establece una correspondencia entre un nombre de dominio completamente cualificado (FQDN) y una dirección IP versión 4.

    
...
ns1.deaw.es.   IN A 192.168.1.20
ns2.deaw.es.   IN A 192.168.1.21
natos.deaw.es. IN A 192.168.1.22
...

  • Registro CNAME (Canonical Name)

    Permite crear alias para nombres de dominio especificados en registros A. Importante: un registro CNAME siempre debe apuntar a otro nombre, nunca directamente a una dirección IP.

    
...
natos.deaw.es. IN A 192.168.1.22
www.deaw.es.   IN CNAME natos.deaw.es.
ftp.deaw.es.   IN CNAME natos.deaw.es.
...

  • Un registro CNAME también puede apuntar a un nombre de otro dominio externo.

    
...
www.deaw.es. IN CNAME www.google.com.
...

  • Registro MX (Mail Exchange)

    Define los servidores encargados de la entrega de correo en el dominio. El número indica la prioridad: a menor número, mayor prioridad. Deben apuntar a un nombre (registro A), no a una IP. 

    
...
deaw.es. IN MX 10 mail1.deaw.es.
deaw.es. IN MX 20 mail2.deaw.es.

mail1.deaw.es. IN A 192.168.1.100
mail2.deaw.es. IN A 192.168.1.101
...

  • Registro PTR (Pointer Record)

    Establece una correspondencia inversa entre direcciones IP y nombres de dominio. Se utilizan en las zonas de resolución inversa para verificar la identidad de un host.

    Para la red 192.168.1.0/24, la sintaxis completa o abreviada sería:

    
...
20.1.168.192.in-addr.arpa. IN PTR ns1.deaw.es.
21.1.168.192.in-addr.arpa. IN PTR ns2.deaw.es.
...

    O de forma abreviada en el archivo de zona inversa:

    
...
20 IN PTR ns1.deaw.es.
21 IN PTR ns2.deaw.es.
22 IN PTR natos.deaw.es.
...
    Entendiendo la jerarquía de la resolución inversa
    • .arpa (Address and Routing Parameter Area)
      Es un dominio de nivel superior (TLD) de infraestructura. A diferencia de los .com o .es, no es para sitios web, sino para funciones técnicas internas del protocolo de Internet.
    • .in-addr
      Es el subdominio dentro de .arpa encargado de gestionar las direcciones IPv4. (Para IPv6 se utiliza .ip6.arpa).
    • ¿Por qué la IP aparece al revés? (Ej: 20.1.168.192)
      Porque el DNS lee los nombres de derecha a izquierda (de lo más general a lo más específico). En una IP, la red es lo más general (192) y el host lo más específico (.20). Al invertirla, el DNS puede buscar primero la red y finalmente el equipo individual, manteniendo la misma lógica de jerarquía que un dominio normal.
    • El punto final (.)
      En DNS, este punto representa la raíz (Root). Indica que el nombre es un FQDN (nombre completo) y que el servidor no debe añadir nada más al final para intentar resolverlo.
    • Importancia en el correo electrónico
      Muchos servidores de correo rechazan mensajes si la IP de origen no tiene un registro PTR válido. Se utiliza para confirmar que el servidor que envía el correo es realmente quien dice ser (evitando el SPAM).

  • Registro TXT (Text)

    Permite asociar texto arbitrario a un dominio. Hoy se usa principalmente para seguridad en el correo (SPF, DKIM) y validación de propiedad de servicios.

    
...
@ IN TXT "Descripción: Servidor maestro de la red DEAW"
@ IN TXT "v=spf1 ip4:192.168.1.100 -all"

Tipos de servidores DNS

Servidor maestro o primario

Un servidor maestro o primario, define una o varias zonas de las que es autorizado. Sus archivos de zona son de lectura y escritura y es en ellos donde el administrador del servidor añade, modifica o elimina nombres de dominio.

  • Si un cliente DNS u otro servidor DNS le pregunta por algún nombre de dominio para el que es autorizado, consulta con los ficheros de zona y responde a la pregunta.
  • Si un cliente DNS u otro servidor DNS le pregunta por algún nombre de dominio para el que no es autorizado, tendrá que preguntar a otros servidores DNS o responder que no conoce la respuesta.

Servidor esclavo o secundario

Un servidor esclavo o secundario define una o varias zonas para las que es autorizado. La diferencia con respecto a un servidor maestro es que los ficheros de zona los obtiene de otro servidor autorizado para la zona, normalmente, de un servidor maestro mediante un procedimiento denominado transferencia de zona. Los ficheros de zona de los servidores esclavos son de solo lectura y por lo tanto, el administrador no tiene que editarlos. La modificación de los archivos de zona debe realizarla el servidor maestro que transfiere la zona.

El funcionamiento de como responden a los clientes DNS o a otros servidores DNS es similar al de un servidor maestro. Un servidor puede ser maestro para una o varias zonas y al mismo tiempo ser esclavo para otras.

Pueden existir varios servidores esclavos para una misma zona. Las razones para esto suelen ser:

  • Reducir y repartir la carga entre varios servidores DNS.
  • Favorecer la tolerancia a fallos.
  • Ofrecer mayor rapidez.

Lo ideal es que los servidores DNS para una misma zona estén ubicados en redes y localizaciones diferentes para evitar que, si ocurre algún problema no les afecte simultáneamente y deje sin servicio de resolución a los nombres de esa zona.

Servidor caché

Los servidores DNS se configuran como servidores cache para mejorar los tiempos de respuesta de las consultas, reducir la carga de los equipos y disminuir el tráfico de red.

Cuando un servidor DNS recibe una pregunta sobre un dominio para el cual no es autorizado, es decir, de un nombre del cual no tiene información, puede preguntar, si así está configurado, a otros servidores para obtener la respuesta. Si el servidor actúa como cache, guarda durante un tiempo (TTL: Time To Live) las respuestas a las últimas preguntas que ha realizado a otros servidores DNS. Cada vez que un cliente DNS u otro servidor DNS le formula una pregunta, comprueba si tiene la respuesta en su memoria cache, si la tiene, no tendrá que preguntar a otro servidor DNS por la pregunta.

Un servidor DNS es solo cache (cache only server) cuando:

  • No tiene autoridad sobre ninguna zona.
  • Pregunta a otros servidores DNS para resolver las preguntas de los clientes DNS y las guarda en su memoria cache.

En el siguiente gráfico se explica como dos clientes DNS hacen preguntas a un mismo servidor DNS que es autorizado para algunas zonas y además actúa como caché.

Servidor forwarder (reenviador)

Cuando a un servidor DNS se le hace una pregunta sobre un nombre de dominio del que no dispone información (no es autorizado), este puede preguntar a otros servidores DNS. Simplificando, existen dos formas de procesar las consultas:

  • El servidor DNS procesa la consulta preguntando a diversos servidores DNS y empezando por los servidores DNS raíz. Consulta iterativa.
  • El servidor DNS reenvía la consulta a otro servidor DNS, denominado reenviador (forwarder), para que se encargue de resolverla. Consulta recursiva.

Visto lo anterior, un reenviador (forwarder) es un servidor DNS que otros servidores DNS designan para reenviarle consultas. Son utilizados para minimizar las consultas y el tráfico de peticiones DNS desde una red hacia Internet. Además permiten a los equipos locales utilizar su cache DNS para minimizar los tiempos de respuesta.

Servidor sólo autorizado

Un Servidor solo autorizado (authoritative only) es aquel que es autorizado para una o varias zonas como servidor maestro y/o esclavo y no responde a preguntas que no sean relativas a sus zonas. Es decir, no tiene activada la recursividad, no es reenviador y no actúa como cache.

Servidores raíz

En Internet existen un conjunto de servidores DNS autorizados para el dominio raíz ., conocidos como servidores raíz (root servers). Contienen el fichero de la zona . que contiene información sobre los servidores DNS autorizados para cada uno de los dominios TLD.

Los servidores raíz son una parte fundamental de Internet, son el primer paso en la traducción (resolución) de los nombres de host en direcciones IP, que se utilizan en la comunicación entre los hosts de Internet. Son claves en el proceso de resolución de nombres de dominio en Internet, y deben de ser conocidos por todos los servidores DNS que respondan a preguntas sobre nombres para los que no son autorizados.

Existen 13 servidores raíz en toda Internet y cada uno de ellos tiene múltiples copias distribuidas por todo el mundo, es decir, que físicamente no solo son 13 servidores. Cada conjunto de copias de uno de los 13 servidores se identifica por una misma IP. Cuando un cliente realiza una pregunta a una IP de un servidor raíz, los routers de Internet encaminan la pregunta hacia la copia más cercana mediante un procedimiento denominado anycasting.

Los nombres de los servidores raíz son de la forma letra.root-servers.net, donde letra va desde la A a la M.

DNS de Horizonte Dividido (Split-Horizon)

Es una configuración donde un mismo servidor DNS ofrece respuestas diferentes dependiendo de quién pregunte (desde dónde venga la IP del cliente). Por ejemplo:

  • Si un empleado pregunta desde la red interna de la empresa, el DNS le da la IP privada del servidor.
  • Si un usuario pregunta desde Internet, el DNS le da la IP pública del mismo servidor.

Maestro Oculto (Stealth Master)

Es un servidor maestro que contiene la zona original pero no aparece listado en los registros NS del dominio. Su existencia es secreta para el público. Solo los servidores esclavos saben quién es para poder copiar la zona. Esto añade una capa extra de seguridad, ya que los atacantes no saben qué servidor es el que realmente permite editar los datos.

Tipos de consultas: recursivas e iterativas

Consultas recursivas

Una consulta recursiva es aquella en la que el servidor DNS da una respuesta completa o exacta. Pueden darse tres tipos de respuesta:

  • Positivas: se devuelve información sobre el dominio consultado
  • Negativas: no se puede resolver el nombre de dominio
  • Error: debido a un fallo en la red

Consultas iterativas

Una consulta iterativa es aquella en la que el servidor DNS proporciona una respuesta parcial. Existen cuatro posibles respuestas:

  • Positivas: se devuelve información sobre el dominio consultado
  • Negativas: no se puede resolver el nombre de dominio
  • Referencia: el servidor DNS indica a otros servidores a los que se le puede consultar para resolver la pregunta
  • Error: debido a un fallo en la red

Resolución inversa

La resolución inversa consiste en obtener información de un nombre de dominio preguntando por la dirección IP en vez de preguntar por el nombre de domino como hemos explicado en apartados anteriores.

Mapeo de direcciones y el dominio arpa

El funcionamiento de la resolución de direcciones IP es igual al de la resolución de nombres de dominio. Las direcciones IP se tratan como nombres que cuelgan del dominio in-addr.arpa para las direcciones IPv4, y del dominio ip6.arpa para las direcciones IPv6.

Cuando usamos una dirección IP, por ejemplo 192.168.1.21, para realizar una pregunta DNS inversa, en realidad estamos preguntando por el nombre de dominio 21.1.168.192.in-addr.arpa. La estructura jerárquica de la dirección IP, tratada como nombre de dominio, es de derecha a izquierda, comenzando por el dominio in-addr.arpa.

.arpa (Address and Routing Parameter Area) es un dominio de nivel superior genérico utilizado sólo para la infraestructura de Internet. Los subdominios de .arpa o dominios de segundo nivel «in-addr.arpa» e «ip6.arpa» son usados por los servidores DNS inversos para la obtención de direcciones IPv4 e IPv6 respectivamente.

Cuando mapeamos una dirección IP estamos asociando la dirección IP al nombre en el dominio .arpa. Por ejemplo la dirección 192.168.1.21 es mapeada al nombre 21.1.168.192.in-addr.arpa.

Zonas de resolución inversa

Los servidores DNS almacenan zonas de resolución inversa con registros de recursos (RR) que asocien nombres de dominio con direcciones IP. Las zonas de resolución inversa pueden ser maestras o primarias y esclavas o secundarias.

Las zonas de resolución directa e inversa son independientes y es responsabilidad de los administradores de los servidores DNS que dichas zonas contengan información coherente y que no existan discrepancias.

No es obligatorio que la entidad que administra una zona de resolución directa de un dominio tenga que administrar la zona de resolución inversa que se corresponda con las direcciones IPs asociadas a dicho dominio.


...
deaw.es.       IN NS ns1.deaw.es.
ns1.deaw.es.   IN A  192.168.1.20
natos.deaw.es. IN A  192.168.1.21
waor.deaw.es.  IN A  192.168.1.22
altea.deaw.es. IN A  192.168.1.23
www.deaw.es.   IN CNAME natos.deaw.es.
ftp.deaw.es.   IN CNAME waor.deaw.es.
...

Proceso de resolución

El proceso de resolución inversa es similar al de resolución directa. Las direcciones IP se tratan como nombres de dominio. Por lo tanto, existen consultas recursivas, iterativas, cache, TTL...

Por ejemplo, si un cliente DNS realiza una consulta recursiva de la IP 192.168.1.21 a un servidor DNS, éste, si no lo tiene en cache, iniciará una serie de consultas iterativas a los servidores DNS raíz, a los servidores autorizados para el dominio 192.in-addr.arpa y así sucesivamente.

Herramientas

Nslookup

Es un programa para consultar servidores DNS. Se utiliza para saber si un servidor DNS resuelve correctamente los nombres DNS y las direcciones IP, para solucionar problemas frecuentes de los servidores DNS o, para diagnosticar problemas ocasionales de configuración en los servidores DNS.

Con nslookup podemos obtener la dirección IP asociada a un nombre DNS y viceversa, además, podemos preguntar a los servidores de nombres información relativa a los registros de recursos (RR) de la/s zona/s de las que son autorizados.

nslookup se usa de dos modos: interactivo y no interactivo. El modo interactivo permite al usuario consultar los servidores DNS para obtener información sobre varios hosts y dominios o para listar los hosts de un dominios. El modo no interactivo se usa para presentar solo el nombre y la información solicitada para un host o nombre DNS.

Este comando funciona tanto en sistemas operativos UNIX/Linux como en Windows. En su momento se trató a nslookup como una aplicación “deprecated” u obsoleta, pero a día de hoy parece que ha vuelto a considerarse apta para su uso normal.

Análisis del comando: nslookup cisco.com
  • Servidor Local (Server: 127.0.0.53)
    Indica qué servidor DNS ha utilizado tu equipo para resolver la consulta. La IP 127.0.0.53 es el resolvedor local por defecto en sistemas Linux modernos (systemd-resolved). El #53 indica que la consulta se hizo a través del puerto 53, el estándar para DNS.
  • Respuesta no autoritativa (Non-authoritative answer)
    Este es un concepto clave. Significa que el servidor que te ha respondido (tu servidor local) no es el dueño original de los registros de Cisco. Ha buscado la información en su memoria caché o ha preguntado a otros servidores, pero la respuesta es "de segunda mano".
  • Nombre y Dirección (IPv4)
    Address: 72.163.4.185
    Muestra la dirección IPv4 asociada al dominio. Es el resultado del Registro A.
  • Nombre y Dirección (IPv6)
    Address: 2001:420:1101:1::185
    Muestra la dirección IPv6 asociada al dominio. Es el resultado del Registro AAAA.
Análisis del comando: nslookup -type=ns cisco.com
  • El parámetro -type=ns
    Le indica a nslookup que no queremos buscar la IP del dominio, sino específicamente sus servidores de nombres (Name Servers). Estos son los servidores que tienen la autoridad para gestionar y responder por cualquier subdominio o registro de cisco.com.
  • Servidor y Dirección (127.0.0.53#53)
    Al igual que en la consulta anterior, el sistema utiliza el resolvedor local de Linux (systemd-resolved) en el puerto estándar 53 para buscar la información.
  • Respuesta no autoritativa (Non-authoritative answer)
    Significa que nuestro servidor local ha obtenido esta lista de servidores desde su memoria caché (información que guardó de consultas previas) en lugar de ir a preguntar directamente a los servidores raíz en este preciso momento.
  • Lista de Nameservers
    Se muestran los 5 servidores autorizados para Cisco. Se observa una mezcla entre servidores propios (ns1, ns2, ns3.cisco.com) y servidores de Akamai (akam.net), lo que garantiza que el DNS sea rápido y resistente a ataques o caídas globales.
  • Authoritative answers can be found from:
    Esta sección (aunque aparece vacía en la captura) indica que si quisiéramos una respuesta "100% oficial" y no de caché, deberíamos dirigir nuestra pregunta a cualquiera de los servidores que aparecen en la lista de resultados.
Análisis de nslookup en modo interactivo
  • Modo Interactivo
    Al ejecutar nslookup sin argumentos, entramos en un subentorno (indicado por el símbolo >). Esto permite realizar varias consultas seguidas o cambiar configuraciones sin tener que volver a escribir el comando inicial.
  • Comando set type=ns
    Dentro del modo interactivo, usamos set type para definir qué clase de registro buscamos. En este caso, configuramos la herramienta para que solo devuelva los Name Servers (servidores de nombres). Otros valores comunes podrían ser set type=mx para correo o set type=a para direcciones IP.
  • Consulta: cisco.com
    Una vez configurado el tipo, simplemente escribimos el nombre del dominio. El sistema utiliza el servidor local (127.0.0.53) para obtener la información.
  • Interpretación de la Respuesta
    Al igual que en el modo de una sola línea, obtenemos una Non-authoritative answer (respuesta no autoritativa). Esto nos recuerda que los datos provienen de la memoria caché de nuestro resolvedor local y no directamente de los servidores de Cisco.
  • Servidores Encontrados
    Se listan los 5 servidores responsables de la zona cisco.com. El hecho de que se repitan los mismos servidores que en el comando host confirma que la base de datos DNS es consistente independientemente de la herramienta que utilicemos.

Dig (Domain Information Groper)

Es la herramienta de referencia para administradores de sistemas y profesionales de red a la hora de realizar diagnósticos y consultas a servidores DNS. A diferencia de otras utilidades, dig ofrece una visión profunda y técnica de las respuestas del servidor.

  • Estándar Profesional: Es considerado el sucesor de herramientas clásicas como nslookup, siendo el estándar actual para resolver problemas de red gracias a su precisión y claridad.
  • Información Exhaustiva: Proporciona una respuesta técnica completa que incluye secciones de encabezado, banderas de estado (flags), secciones de autoridad y tiempos de respuesta detallados.
  • Versatilidad de Uso: Aunque su uso más común es a través de la interfaz de línea de comandos (CLI), destaca por su capacidad de operar por lotes, permitiendo leer múltiples consultas desde un archivo de texto externo.
  • Multiplataforma: Es una herramienta nativa de sistemas UNIX/Linux, pero su utilidad ha hecho que esté disponible y sea ampliamente utilizada también en entornos Windows.

Nota: La principal ventaja de dig es que muestra la respuesta del servidor DNS exactamente como se recibe, sin procesar ni simplificar, lo que facilita la detección de errores de configuración.

Análisis avanzado con el comando: dig cisco.com a
  • Encabezado (HEADER) y Estado
    status: NOERROR indica que la consulta fue exitosa. Las flags nos dan información clave:
    • qr (Query Response): Es una respuesta.
    • rd (Recursion Desired): Pedimos que el servidor buscara por nosotros.
    • ra (Recursion Available): El servidor DNS admite búsquedas recursivas. Nota: Al no aparecer la bandera aa, sabemos que la respuesta no es autoritativa (viene de caché).
  • Sección de Pregunta (QUESTION SECTION)
    ;cisco.com. IN A
    Aquí se muestra exactamente qué le preguntamos al servidor: buscamos el registro de clase Internet (IN) del tipo A para el dominio cisco.com.
  • Sección de Respuesta (ANSWER SECTION)
    cisco.com. 300 IN A 72.163.4.185
    Este es el resultado final. El número 300 es el TTL (tiempo de vida en segundos), lo que significa que esta respuesta se guardará en caché durante 5 minutos antes de volver a preguntar al servidor maestro.
  • Estadísticas de la consulta
    • Query time: 144 msec (tiempo que tardó en llegar la respuesta).
    • SERVER: 127.0.0.53#53 (indica que respondió el resolvedor local del sistema).
    • WHEN: La fecha y hora exacta de la consulta (19 de enero de 2026).
Análisis de servidores de correo con: dig cisco.com mx
  • Registro MX (Mail Exchange)
    El parámetro mx indica que estamos consultando los "intercambiadores de correo". Estos registros le dicen al mundo a qué servidores deben enviarse los emails dirigidos a @cisco.com.
  • Sección de Respuesta (ANSWER SECTION)
    Se muestran 3 servidores de correo. La estructura es: [Dominio] [TTL] [Clase] [Tipo] [Prioridad] [Nombre del servidor].
    • El servidor alln-mx-01 tiene prioridad 10 (es el principal).
    • Los servidores con 20 y 30 son secundarios y actúan como respaldo.
  • El TTL (463)
    En esta consulta, el tiempo de vida (TTL) es de 463 segundos. Es un número "extraño" porque es una cuenta atrás; originalmente pudo ser de 600 segundos, pero como la respuesta viene de la caché, nos indica que en poco más de 7 minutos la información caducará y el servidor tendrá que volver a preguntar.
  • Query time: 0 msec
    Este es un dato revelador. Un tiempo de respuesta de 0 milisegundos significa que la respuesta ha sido instantánea porque ya estaba guardada en la memoria RAM del resolvedor local (127.0.0.53). No ha hecho falta salir a Internet para obtenerla.
  • Flags: qr rd ra
    Nuevamente, vemos que no aparece la flag aa (Authoritative Answer). Esto confirma que, aunque la información es correcta, nuestro servidor nos la está dando "de segunda mano" desde su caché.

Host

host es una herramienta CLI sencilla y fácil de usar para realizar consultas DNS, que traducen nombres de dominio a direcciones IP y viceversa. También se utiliza para consultar los registros DNS de las zonas que almacenan los servidores DNS, probar y validar el servidor DNS y la conectividad a Internet, registros de correo no deseado y listas negras, diagnóstico de problemas en el servidor DNS...

Análisis de la consulta general: host cisco.com
  • Registro A (IPv4)
    cisco.com has address 72.163.4.185
    Indica la dirección IP física (versión 4) del servidor web. Es el dato que utiliza el navegador para cargar la página.
  • Registro AAAA (IPv6)
    cisco.com has IPv6 address 2001:420:1101:1::185
    Es el equivalente al registro A pero para el protocolo IPv6. Las empresas grandes como Cisco siempre ofrecen ambas para asegurar conectividad global moderna.
  • Registros MX (Mail Exchange)
    mail is handled by 10 alln-mx-01.cisco.com...
    Estos registros indican qué servidores reciben los correos electrónicos enviados a @cisco.com.
  • Prioridad en el Correo
    Los números 10, 20 y 30 indican el orden de preferencia. El servidor con el número más bajo (10: alln-mx-01) es el principal. Si este falla, el correo se enviará al siguiente (20) y así sucesivamente.
  • Balanceo y Redundancia
    Al mostrar múltiples registros MX y direcciones IP, se demuestra que el dominio está preparado para la alta disponibilidad: si un centro de datos cae, los otros registros permiten que los servicios sigan funcionando.
Análisis de la resolución inversa: host 72.163.4.185
  • Consulta Inversa Automática
    Cuando ejecutamos host seguido de una dirección IP, el comando entiende que no buscamos una IP, sino que queremos hacer el camino contrario: obtener el nombre asociado a esa dirección (proceso conocido como Reverse DNS Lookup).
  • 185.4.163.72.in-addr.arpa.
    Como se observa en la respuesta, el sistema ha invertido automáticamente la IP 72.163.4.185 y le ha añadido el sufijo .in-addr.arpa. para poder buscarla en la jerarquía DNS mundial.
  • domain name pointer (PTR)
    La respuesta indica que ha encontrado un "puntero de nombre de dominio". Este es el nombre técnico del Registro PTR que vincula esa IP específica con un nombre concreto.
  • Resultado: redirect-ns.cisco.com.
    Este es el nombre oficial (FQDN) configurado para esa IP. Es común que las grandes empresas usen nombres técnicos como redirect-ns (Redirect Name Server) para sus IPs públicas de infraestructura.
  • ¿Para qué sirve esta comprobación?
    Sirve para verificar la identidad. Si un servidor dice ser de Cisco pero su resolución inversa apunta a un dominio desconocido o sospechoso, otros servidores en Internet podrían bloquearlo por seguridad o marcar su correo como SPAM.
Análisis del comando: host -t NS cisco.com
  • El parámetro -t NS
    Indica que estamos filtrando la búsqueda para que solo nos muestre los registros de tipo Name Server. Estos registros definen qué servidores DNS son los "dueños" o autorizados para responder por el dominio cisco.com.
  • Servidores Propios (ns1, ns2, ns3.cisco.com)
    Cisco utiliza sus propios servidores DNS para gestionar parte de su tráfico. Al tener varios (1, 2 y 3), se aseguran de que si uno falla, los otros puedan seguir respondiendo (redundancia).
  • Servidores Externos (akam.net)
    Observamos que Cisco también delega su autoridad en Akamai (akam.net), una de las redes de distribución de contenido (CDN) más grandes del mundo. Esto sirve para que las consultas DNS se respondan de forma mucho más rápida desde cualquier parte del planeta.
  • ¿Por qué aparecen 5 servidores?
    Por tolerancia a fallos. Si un atacante intentara tirar el servicio DNS de Cisco, tendría que tumbar los servidores en sus centros de datos y también los de Akamai simultáneamente, lo cual es extremadamente difícil.
  • El proceso de resolución
    Cuando alguien busca www.cisco.com, los servidores raíz y los TLD (.com) le dirán: "Yo no sé la IP, pero pregúntale a cualquiera de estos 5 servidores de la lista, ellos tienen la verdad oficial".

Whois

Aunque no es una herramienta de diagnóstico DNS si que nos ofrece información sobre el registro del dominio.

whois es un protocolo que permite realizar consultas a bases de datos que contienen información; del usuario, empresa u organización que registra un nombre de dominio y/o una dirección IP en Internet. El protocolo whois se encapsula en TCP y solo especifica el intercambio de peticiones y respuestas, no el formato de datos a intercambiar. Por eso, los resultados de las consultas whois pueden variar dependiendo de la base de datos whois a la que se pregunte.
